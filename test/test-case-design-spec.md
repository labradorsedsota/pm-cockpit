## 文档定位

本文档是测试规范三层链条的**第二层（设计层）**，解决"如何设计高质量测试用例"。

| 层级 | 文档 | 职责 |
|------|------|------|
| 指派层 | [test-requirements.md](../skill/references/test-requirements.md) | PM 如何指派测试任务 |
| **设计层** | **本文档** | **QA 如何设计测试用例** |

**工作流：** PM 指派（指派层）→ QA 设计用例（本文档）→ QA 执行测试（执行层）

---

## 背景

Phase 0 及 OMS 项目的测试实践中，测试用例设计依赖 QA 经验，缺乏系统化的设计方法论。回顾分析发现以下设计层缺陷：

| 缺陷 | 影响 | 对应条款 |
|------|------|---------|
| 无边界值测试 | 高缺陷发现率手段缺失，只测"正常值"和"空值" | 条款 D1 |
| 测试间数据污染 | 用例 A 修改数据后影响用例 B 的预期结果 | 条款 D2 |
| 状态转换只测正向 | 只验证合法路径，非法转换未覆盖 | 条款 D3 |

## 条款 D1：边界值分析（BVA）

### D1.1 强制要求

对 PRD 中每个可编辑输入字段，QA 必须识别其边界并设计对应用例。

**适用范围：** 所有接受用户输入的字段（文本框、数字输入、下拉选择、日期选择器等）。

### D1.2 边界识别方法

对每个输入字段，按以下步骤分析：

1. **确定有效范围** — 从 PRD 或技术实现中提取约束（如：数量 ≥ 1，单价 > 0，名称非空且 ≤ 50 字符）
2. **标记边界点** — 有效范围的最小值、最大值
3. **取边界及相邻值** — 最小值、最小值-1、最小值+1、最大值、最大值+1、最大值-1

### D1.3 各字段类型的 BVA 模板

**数值字段：**

| 边界点 | 测试值 | 预期行为 |
|--------|--------|---------|
| 最小有效值 | min | 接受 |
| 低于最小值 | min - 1 | 拒绝 + 错误提示 |
| 最大有效值 | max | 接受 |
| 超过最大值 | max + 1 | 拒绝 + 错误提示 |
| 零值 | 0 | 取决于业务规则 |
| 负数 | -1 | 通常拒绝 |

**文本字段：**

| 边界点 | 测试值 | 预期行为 |
|--------|--------|---------|
| 空字符串 | "" | 必填字段拒绝 |
| 单字符 | "a" | 取决于最小长度 |
| 最大长度 | "a" × maxLen | 接受 |
| 超过最大长度 | "a" × (maxLen + 1) | 截断或拒绝 |
| 特殊字符 | `<script>`, `'; DROP TABLE`, `你好🎯` | 不应导致异常 |

**日期字段：**

| 边界点 | 测试值 | 预期行为 |
|--------|--------|---------|
| 今天 | 当日日期 | 通常接受 |
| 过去日期 | 昨天 | 取决于业务规则 |
| 远未来 | 2099-12-31 | 取决于业务规则 |
| 无效日期 | 2025-02-30 | 拒绝 |

### D1.4 用例文档标注

BVA 用例在 test case 文档中必须标注 `[BVA]` 标签和目标边界，格式：

```
| 项 | 值 |
|---|---|
| mosstid | `{项目}-{app}-v1-L3.X-001` |
| 设计技术 | BVA — 数量字段下界 |
| 目标边界 | 数量 = 0（有效最小值为 1） |
```

### D1.5 覆盖率要求

- 每个可编辑数值字段：至少 2 条 BVA 用例（有效边界 + 无效边界各 1 条）
- 每个必填文本字段：至少 1 条 BVA 用例（空值/特殊字符）
- BVA 用例通常归入 L3（边缘场景），除非 PRD 明确定义了约束（则归入 L2）

## 条款 D2：数据隔离声明

### D2.1 强制要求

每条测试用例必须声明其**数据状态依赖**，明确本条用例启动时需要什么数据状态。

### D2.2 数据依赖类型

| 依赖类型 | 含义 | Pre-flight 动作 |
|----------|------|----------------|
| `SEED` | 需要完整种子数据（初始状态） | 清空存储 + 重新加载页面触发种子注入 |
| `EMPTY` | 需要空数据 | 清空存储 + 注入空数组阻止种子注入 |
| `CUSTOM` | 需要特定数据状态 | 清空存储 + 通过 JS/API 注入自定义数据 |
| `ANY` | 对数据状态无依赖 | 无需额外清理（仍执行标准 Pre-flight） |

### D2.3 用例文档标注

在 test case 文档的元信息表中增加 `数据依赖` 字段：

```
| 项 | 值 |
|---|---|
| mosstid | `oms-oms-v1-L2.6-001` |
| 数据依赖 | SEED — 需要种子数据完整（8条订单含各状态） |
| 冲突标记 | ⚠ 与 L1.3（修改订单状态）冲突，必须在 L1.3 之后重置 |
```

### D2.4 冲突检测规则

QA 在完成全部用例设计后，必须执行一轮冲突扫描：

1. 列出所有**写操作用例**（增删改状态的用例）
2. 列出所有**依赖特定数据的读操作用例**（筛选、统计、列表验证）
3. 检查：如果按 L1 → L2 → L3 顺序执行，写操作用例是否会破坏后续读操作用例的数据依赖
4. 存在冲突的用例对必须标注 `⚠ 冲突标记`，并在 Pre-flight 中显式清理

### D2.5 与执行规范的衔接

本条款声明数据依赖（设计阶段）。实际的清理执行由 [mano-cua-execution-spec.md 条款 1c](./mano-cua-execution-spec.md#条款-1c数据隔离保障v10-新增) 保障。

两者的职责划分：
- **设计规范（本文档）**：QA 声明"需要什么数据状态" + 标注冲突
- **执行规范（条款 1c）**：执行时按声明执行对应清理脚本

## 条款 D3：状态转换否定路径

### D3.1 强制要求

当 PRD 定义了实体状态机（如订单状态：待确认 → 已确认 → 生产中 → 已发货 → 已完成）时，测试用例必须同时覆盖：
- **正向路径**（合法转换）— 已有 L1 用例覆盖
- **否定路径**（非法转换）— 本条款新增要求

### D3.2 否定路径设计方法

1. **绘制状态转换矩阵**

以 OMS 订单状态为例：

| 当前状态 ↓ \ 目标状态 → | 待确认 | 已确认 | 生产中 | 已发货 | 已完成 |
|--------------------------|--------|--------|--------|--------|--------|
| 待确认 | - | ✅ | ✗ | ✗ | ✗ |
| 已确认 | ✗ | - | ✅ | ✗ | ✗ |
| 生产中 | ✗ | ✗ | - | ✅ | ✗ |
| 已发货 | ✗ | ✗ | ✗ | - | ✅ |
| 已完成 | ✗ | ✗ | ✗ | ✗ | - |

✅ = 合法转换（正向用例覆盖）
✗ = 非法转换（否定用例覆盖）

2. **选择否定用例**

不需要覆盖所有 ✗ 格子。按以下优先级选择：
- **终态无操作**（必测）：已完成状态不应显示推进按钮
- **回退测试**（必测）：任意非初始状态不应有回退按钮或回退操作
- **跳跃测试**（选测）：如从"待确认"能否直接跳到"已发货"

3. **最低覆盖要求**

每个状态机至少 2 条否定用例：
- 1 条终态验证（终态无法继续推进）
- 1 条回退验证（中间状态无法回退到前一状态）

### D3.3 用例文档标注

否定路径用例在 test case 文档中必须标注 `[否定路径]` 标签：

```
| 项 | 值 |
|---|---|
| mosstid | `oms-oms-v1-L2.X-001` |
| 设计技术 | 状态转换否定路径 — 回退测试 |
| 测试目标 | 验证"已确认"状态无法回退到"待确认" |
```

### D3.4 与 L1 正向用例的关系

- L1 正向用例验证"合法转换可以完成"
- L2/L3 否定用例验证"非法转换被阻止"
- 两者共同构成状态机的完整覆盖

## 附录：test case 模板（含新增字段）

在现有 test case 模板基础上，增加以下字段：

```markdown
### L{层级}.{序号} {用例名称}

| 项 | 值 |
|---|---|
| mosstid | `{项目}-{app}-{版本}-{测试点}-{序号}` |
| 关联 AC | AC-X.X.X |
| 设计技术 | PRD 追溯 / BVA / 状态转换否定路径 |
| 数据依赖 | SEED / EMPTY / CUSTOM / ANY |
| 冲突标记 | 无 / ⚠ 与 L{X}.{Y} 冲突（原因） |
| 前置策略 | 策略一 / 策略二 |
| fixture | 文件名或"无" |

**Pre-flight：** ...
**任务描述：** ...
**Expected Results（逐条）：** ...
```
