## BLOCKED 强制诊断要求

> 来源：OMS 项目回顾 — L3.2/L3.3/L3.6 标记 BLOCKED 后无诊断，悬空 16 小时

QA 标记测试点为 BLOCKED 时，**禁止只标状态跳过**，必须同时附带：

1. **失败现象描述** — 具体报错信息或截图
2. **根因判断** — 初步分析为什么失败（环境、权限、依赖、工具）
3. **至少 2 种解决方案** — 每种方案附 tradeoff 说明和预估修复时间

格式：
```
⚠️ BLOCKED: {测试点ID}
现象：{具体描述}
根因：{初步判断}
方案 A：{描述} — 优点：{} / 缺点：{} / 预估：{时间}
方案 B：{描述} — 优点：{} / 缺点：{} / 预估：{时间}
建议采用：方案 {X}，原因：{}
@Pichai
```

**PM 指派测试时必须显式传达此要求。**

## Fixture 要求

- 测试前全部准备就绪
- 存放位置：`test/fixtures/{app}/`
- 生成责任：Moss（QA 最清楚需要什么输入）
- 内容须覆盖 PRD 要求的测试边界

## mosstid 要求

- 每条测试记录 mosstid（轨迹 ID）
- 格式：`{项目小写}-{应用小写}-{版本}-{测试点}-{序号}`
- 嵌入 mano-cua session 水印
