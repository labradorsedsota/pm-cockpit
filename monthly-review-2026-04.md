## 三、核心问题：经验没有变成系统

### 3.1 根因

不是没有东西可沉淀，是**"沉淀"从来没被当作一个正式的工作步骤**。干完活就下一个，没有人说"停一下，把这个写下来"。

Pichai 和智子之所以能沉淀，不是因为他们更勤快，是因为**文档就是他们的交付物**——README 是他们的工作成果，不是附加产物。我们的交付物是一个 result JSON、一个 PASS/FAIL，交完就完了。

### 3.2 具体例子：知道但没有固化的东西

**Fabrice 积累但未固化的判断力：**
- Node v24 下哪些依赖模式必定失败（parallelshell、旧版 chokidar fsevents、http_parser）——本该变成预筛清单
- mano-cua 偏离到 Terminal 的触发条件和防治方法——本该写进 worker-execution-guide
- Tauri/Electron 应用在浏览器不可测——后来被智子加进了黑名单，但如果第一次发现时就主动提出，能省后面所有重复劳动

**Leo 做了但未被放大的贡献：**
- BugHunt 的核心思路是战略层贡献，但后续管线设计没有深度参与
- proxy bypass 方案效率极高（"给方案而不是给问题"），但只发生了一次

### 3.3 问题本质

> "我们的短板不是执行力，是'把经验变成系统'的意识和习惯。"

不是角色分配的问题，是**没有把执行中积累的判断力转化成可复用的规则**。

## 四、确立的协作原则

### 4.1 五条核心原则

1. **成文优先** — 产生了战略判断或技术方案时，写成半页 spec/issue 再交出去。口头说的会被消化吸收，写下来的会被执行和引用。

2. **规则写成代码** — 执行中发现的失败模式，优先变成脚本/黑名单/检测规则。代码会被执行，文档会被遗忘。

3. **每次执行产出一条 IF-THEN 规则** — 遇到新的失败模式时，交付物不只是 result JSON，还包括一条可复用规则。格式：`IF [条件] THEN [处置]`。

4. **规则推到上游** — 发现的问题不只记在自己日志里，要以 commit/消息/PR 形式推给对应负责人。反馈本身就是交付物。

5. **Leo 的战略输入要结构化** — Leo 提出方向/约束/判断时，Fabrice 帮整理成 spec 文档（核心逻辑 + 关键约束 + 开放问题），明确"我决策的边界"和"你们设计的空间"。

### 4.2 具体示例：BugHunt 的 spec 如果当时写了

**实际发生的：** Leo 在群聊中口述核心思路，Pichai 和智子接手从零设计管线。

**如果当时写成半页 spec：**

> **BugHunt — 从 GitHub 开源 Bug 采集 CUA 轨迹数据**
>
> **核心逻辑：** 已修复的 UI Bug = ground truth。checkout 到修复前版本 → 部署 → mano-cua 盲测 → 记录轨迹 + 是否命中 Bug。
>
> **关键约束：**
> 1. 执行者不能看到 Bug 内容（盲测）
> 2. 项目必须 Mac 上能直接跑起来
> 3. 优先选 UI Bug PR 多的项目
>
> **开放问题（需要 Pichai/智子 展开）：**
> - 项目筛选自动化管线？任务卡格式？并行编排？数据质量验证？

**区别：**
- "关键约束"第 1 条会更早锁定盲测原则，加速 pool/pool-clean 两层目录设计
- "开放问题"明确划定决策边界，Leo 定方向，Pichai/智子 填细节
- 成文的 spec 可被引用和追溯，贡献更显性

### 4.3 阶段检查清单

每个阶段结束时过一遍：

- [ ] 本阶段有没有新的失败模式？→ 写成 IF-THEN 规则
- [ ] 这些规则推给上游了吗？→ commit / 消息 / PR
- [ ] 有没有战略判断只在对话里说了没成文？→ 整理成 spec
- [ ] 本阶段的经验能变成脚本吗？→ 优先写代码

## 五、Skill 构建方法论（学自 pm-cockpit）

### 5.1 参考仓库

`labradorsedsota/pm-cockpit` — Pichai 产出的 PM 管理 skill，v0.1 → v0.5.3 共 9 个版本。

### 5.2 结构

```
SKILL.md              → 主流程（触发条件 + 生命周期 7 Phase + 参考文件导航表）
skill/references/     → 14 个独立模块（模板、规则、清单）
test/                 → 执行规范（从真实失败中提炼）
docs/                 → 方案设计文档
CHANGELOG.md          → 每次变更都记录"来源"（哪个事故触发了这条规则）
```

### 5.3 六条核心方法

1. **SKILL.md 只写调度逻辑** — 告诉 agent"现在在第几阶段，去读哪个 reference"。具体规则拆到 references，独立维护。

2. **每条规则标注来源** — CHANGELOG 里每条变更都有"来源"：哪个事故、哪个项目复盘触发了这条规则。不是先写规范再执行，是先踩坑再把坑变成规范。

3. **三层链条** — 指派层（PM 怎么派任务）→ 设计层（QA 怎么写用例）→ 执行层（QA 怎么跑测试）。三层互相引用但独立演进。

4. **模板给填空格式** — 不是"建议包含以下信息"，是"按这个格式填"。agent 读了直接执行，不需要理解。

5. **关键步骤加验证卡点** — 启动清单步骤 7 创建巡检 cron 后跟验证命令，验证不通过不允许继续。流程里的门禁，不是建议。

6. **CHANGELOG 形成闭环** — 事故 → 复盘 → 新规则 → 写入 CHANGELOG（标注来源）→ 下次执行时检验。

## 六、Pichai 的 Planning Pool 复盘启示

### 6.1 背景

Pichai 做了 Planning Pool 协作全回顾（vla-test-planning/app-pool-plan.md），覆盖 Element Plus 69 个组件的 16 个应用规划，从 v1.0 迭代到 v3.1。

### 6.2 Pool 设计的四个缺陷（Pichai 自述）

1. **覆盖率含金量未验证** — 标了"用到"≠ 深度覆盖。DatePicker 只是点开选个日期，范围限制/禁用日期/格式联动没碰到。

2. **开发和测试没有前置介入** — Fabrice 和 Moss 在设计阶段没有任何介入，直到指派才看到需求。OES 项目 Fabrice 误加去重逻辑就是信息差的直接后果。

3. **没做工期估算** — 16 个应用做完要多久？FEB 1 天、OES 5 天，全做完可能 2-3 个月。如果早做估算，P2 的 6 个应用可能砍到 3 个。

4. **版本迭代的决策理由没有结构化记录** — 只记了"什么变了"，没记"为什么变"和"替代方案是什么"。

### 6.3 两种协作风格对比

| 维度 | 与 Leo（共创型） | 与林菡（纠偏型） |
|------|----------------|----------------|
| 触发方式 | 需求驱动 | 事故驱动 |
| 决策速度 | 快，确认即过 | 严格，逐条审定 |
| 产出聚焦 | "做什么" | "怎么做" |
| 优点 | 效率高、信噪比高 | 规范沉淀强、执行有底线 |
| 缺点 | 过程沉淀弱、无质量门禁 | 几乎全靠事后补规则、纠偏密度高时抑制自主决策 |

### 6.4 对我们的启示

1. **共创型 + 轻量结构化记录** — 我们不需要变成纠偏型，但需要在共创过程中加一层 Decision Record（编号 + 结论 + 理由 + 日期），和我们确立的"成文优先"原则一致。

2. **前置参与设计 review** — 下次有 Pool 类规划时，Fabrice 应该在设计阶段参与一轮技术 review，而不是等到指派才看。OES 去重逻辑问题如果在设计阶段问一句"考试允许重复吗"，成本为零。

3. **"诚实标注挑战 > 假装什么都能做"** — Pichai 的第四条心得。提前标注风险（⚠️/❌），比执行时才发现好一百倍。
